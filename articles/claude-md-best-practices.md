CLAUDE.md の書き方がわからない、テンプレートが欲しい――そんなベストプラクティスを求める声が増えています。`/init` で自動生成したまま放置していませんか？ 実はそれ、Claudeの性能を下げている可能性があります。

## CLAUDE.md とは何か？なぜ重要なのか

CLAUDE.md は、Claude Code がセッション開始時に毎回読み込む設定ファイルです。プロジェクトのルール、コーディング規約、ワークフローを伝える「常駐プロンプト」として機能します。

ここで重要なのは **「毎回読み込まれる」** という点です。CLAUDE.md に書いた内容はすべてコンテキストウィンドウを消費します。冗長な記述は、本来コードの読解や推論に使われるべきトークンを圧迫し、Claudeの応答品質を直接劣化させます。

## 冗長な CLAUDE.md が引き起こす3つの問題

### 1. 指示の希薄化

フロンティアLLMが一貫して従える指示は **150〜200個程度** と言われています。300行を超えるCLAUDE.mdに50個以上のルールを詰め込むと、重要な指示が埋もれ、Claudeは「どれも中途半端に守る」状態に陥ります。

### 2. `/init` のノイズ問題

`/init` コマンドで生成されるCLAUDE.mdは便利な出発点ですが、そのまま使うと問題があります。ディレクトリ構成の冗長な説明（`/components にはコンポーネントが入っています`）や、Claudeが推論できる自明な情報が大量に含まれます。

### 3. コンテキストの浪費

1Mコンテキストが使える時代でも、CLAUDE.mdの肥大化は避けるべきです。トークンは有限のリソースであり、ファイル読み込み・コード解析・推論のために温存すべきです。

## CLAUDE.md 書き方のベストプラクティス：WHY / WHAT / HOW

効果的なCLAUDE.mdは **3つの層** で構成します。

### WHY（なぜこのプロジェクトが存在するか）

2〜3行でプロジェクトの目的を伝えます。Claudeが「何のためのコードか」を理解していれば、設計判断の精度が上がります。

### WHAT（何を守るべきか）

コーディング規約、アーキテクチャ上の制約、チーム固有のルールを記述します。**コードから推論できないもの** だけを書くのがポイントです。

### HOW（どう作業するか）

ビルド・テスト・デプロイのコマンド、ブランチ戦略、PRのルールなど、ワークフローを明記します。

## 悪い例：肥大化した CLAUDE.md

```markdown
# プロジェクト概要
これはNext.jsを使ったWebアプリケーションです。
Next.jsはReactベースのフレームワークで、SSRとSSGをサポートしています。
TypeScriptで書かれており、型安全性を重視しています。

# ディレクトリ構成
- /src - ソースコード
- /src/components - Reactコンポーネント
- /src/components/ui - UIコンポーネント（ボタン、入力フィールドなど）
- /src/components/layout - レイアウトコンポーネント（ヘッダー、フッターなど）
- /src/hooks - カスタムフック
- /src/utils - ユーティリティ関数
- /src/types - 型定義ファイル
- /public - 静的ファイル
- /tests - テストファイル

# コーディングルール
- 変数名はcamelCaseを使ってください
- コンポーネントはPascalCaseで命名してください
- インデントはスペース2つです
- セミコロンは省略しないでください
- 文字列にはシングルクォートを使ってください
- constを優先し、letは必要な場合のみ使ってください
- anyの使用は禁止です
- console.logは本番コードに残さないでください
# ... さらに50行以上のルールが続く
```

**問題点:**
- Next.jsやTypeScriptの一般知識をClaudeに教えている（Claudeは既に知っている）
- ディレクトリ名から自明な説明を羅列している
- ESLintやPrettierで強制できるルールをCLAUDE.mdに書いている
- 120行以上あるのに、プロジェクト固有の情報がほぼない

## 良い例：簡潔で効果的な CLAUDE.md テンプレート

```markdown
## プロジェクト
SaaS型請求書管理サービス。マルチテナント。Stripe連携で決済処理。

## 技術スタック
Next.js 15 / Drizzle ORM / Neon PostgreSQL / NextAuth v5

## コマンド
- `pnpm dev` - 開発サーバー
- `pnpm test` - Vitest実行
- `pnpm db:push` - スキーマ反映
- `pnpm lint` - ESLint + Prettier（コミット前に必ず実行）

## アーキテクチャ上の制約
- DBアクセスは必ず `src/server/` 配下で行う（クライアントから直接アクセスしない）
- テナント分離は RLS で実装済み。新テーブル追加時は必ず tenant_id カラムと RLS ポリシーを設定する
- API Route は `src/app/api/` に配置。認証チェックは `withAuth` ミドルウェアを通す

## 規約
- エラーハンドリング: Result型パターン（`src/lib/result.ts`）を使う。throw しない
- テスト: 新機能には必ずユニットテストを書く。`__tests__/` に配置
- コミット: Conventional Commits 形式（feat: / fix: / chore:）

## 詳細ルール
- コード規約の詳細 → `ai/rules/CODE_STYLE.md`
- Git ワークフロー → `ai/rules/GIT_WORKFLOW.md`
```

**優れている点:**
- 50行以内に収まっている
- Claudeが推論できない **プロジェクト固有の制約** に集中している
- リンターで強制できるルールは書かず、コマンドだけ示している
- 詳細ルールは別ファイルに `@import` で分離している
- コマンドが明確で、Claudeが即座に実行できる

## 「削除してもClaudeが間違わないなら削除」の判断基準

CLAUDE.md を最適化する最強の原則がこれです。具体的な判断フローを示します。

**ステップ1: その指示を消して、Claudeに同じタスクをやらせる**
正しく動くなら、その指示は不要です。

**ステップ2: 以下に該当するなら削除対象**
- フレームワークやライブラリの一般知識（Claudeは既に知っている）
- ディレクトリ名から自明な説明（`/utils にはユーティリティがある`）
- ESLint / Prettier / Biome で強制できるフォーマットルール
- 1つのタスクでしか使わない情報（→ Skills に移動）

**ステップ3: 以下は残すべき**
- プロジェクト固有のアーキテクチャ制約
- チーム独自の命名規則やパターン
- ビルド・テスト・デプロイのコマンド
- Claudeが繰り返し間違えた箇所の修正ルール

## 運用のコツ：失敗駆動で育てる

完璧なCLAUDE.mdを最初から書こうとしないでください。最も効果的なアプローチは **失敗駆動の反復改善** です。

1. **最小限で始める** - 20〜30行のCLAUDE.mdからスタート
2. **Claudeが間違えたら追加する** - 実際のミスをルールとして記録
3. **定期的に棚卸しする** - 月1回、Claudeに「このCLAUDE.mdを最適化して」と依頼
4. **gitで管理する** - チーム全員で改善を積み重ねる

## まとめ

CLAUDE.md の書き方のベストプラクティスをまとめます。

| 原則 | 内容 |
|------|------|
| **300行以内** | ルートは50〜100行。詳細は別ファイルに分離 |
| **削除テスト** | 消してもClaudeが間違わないなら削除 |
| **WHY/WHAT/HOW** | 3層構造で必要十分な情報を伝える |
| **リンターに任せる** | フォーマットルールはCLAUDE.mdに書かない |
| **失敗駆動** | 間違えたときだけルールを追加する |
| **定期メンテ** | 月1回の棚卸しで肥大化を防ぐ |

テンプレートをそのままコピーして使うのではなく、自分のプロジェクトに合わせて削ぎ落とすことが大切です。CLAUDE.md は「Claudeへの指示書」ではなく「Claudeとの共有コンテキスト」。短く、鋭く、プロジェクト固有の情報だけを伝えましょう。
