Claude Codeが遅い・重いと感じたことはありませんか？ トークン消費が異常に増えて、セッション途中でレート制限に到達してしまう。私自身、約1ヶ月かけてこの問題の原因を突き止めました。本記事では、Claude Codeの速度低下とトークン過剰消費の具体的な原因と、実証済みの対策をコード例付きで解説します。

## Claude Codeが遅い・重い3大原因

速度低下の原因は大きく分けて3つあります。どれも再現性が低く、特定に時間がかかるのが厄介です。

### 原因1: CLAUDE.mdの肥大化

CLAUDE.mdはセッション開始時に**毎回まるごと読み込まれます**。200行を超えると、それだけで数千トークンを消費し、指示が長すぎてClaude自身がルールを見落とす現象も発生します。

```markdown
# ❌ 悪い例：CLAUDE.mdに全部詰め込む（300行超え）
## プロジェクト概要
...（50行）
## コーディング規約
...（80行）
## APIドキュメント
...（100行）
## トラブルシューティング
...（70行）
```

**対策：CLAUDE.mdは50行以内に絞り、詳細はSkillファイルに分離する。**

```markdown
# ✅ 良い例：CLAUDE.mdはルーティングだけ（40行）
## 基本原則
- シンプルさ最優先
- 影響を最小化する

## プロジェクト構成
詳細は `ai/rules/PROJECT_ARCHITECTURE.md` を参照

## コード規約
詳細は `ai/rules/REACT_NATIVE_GUIDELINES.md` を参照

## スキル
- デプロイ → `~/.claude/skills/deploy/SKILL.md`
- テスト → `~/.claude/skills/test/SKILL.md`
```

Skillファイルはオンデマンドでロードされるため、毎回のコンテキストを圧迫しません。この変更だけで初期トークン消費が30〜40%削減されたケースが報告されています。

### 原因2: コンテキストウィンドウの肥大化

長時間セッションを続けると、会話履歴がコンテキストウィンドウを圧迫します。70%を超えたあたりから応答速度が目に見えて低下し、ツール呼び出しの精度も落ちます。

```bash
# コンテキスト使用量を確認（Claude Code内で実行）
/context

# 出力例:
# Context usage: 78% (156,000 / 200,000 tokens)
# ⚠️ Warning: Context is getting full. Consider compacting.
# Top consumers:
#   - Tool results: 45%
#   - Conversation history: 30%
#   - System prompt (CLAUDE.md): 15%
#   - Other: 10%
```

**対策：30〜45分ごとに `/compact` を実行し、会話を圧縮する。**

```bash
# 会話履歴を自動要約・圧縮
/compact

# タスクが切り替わったらクリア
/clear
```

2026年3月のアップデート（v2.1.74）では、50Kを超えるツール結果がディスクに永続化されるようになり、コンテキスト消費が自動的に抑制されるようになりました。最新版へのアップデートも効果的です。

### 原因3: claude-memとの競合

claude-memはClaude Codeのフック機構を使って観察結果を記録するプラグインですが、Agent SDKのサブプロセス管理と競合し、プロセスがexit code 1で終了する問題が報告されています（[GitHub Issue #696](https://github.com/thedotmack/claude-mem/issues/696)）。

症状としては：
- セッション開始直後は正常だが、途中から急激に遅くなる
- 観察（observation）が記録されない
- CPUやメモリ使用率が異常に上昇する

**対策：一時的にclaude-memを無効化して、速度が改善するか切り分ける。**

```json
// ~/.claude/settings.json から一時的にclaude-memのhooksを無効化
{
  "hooks": {
    // "post_tool_use": "claude-mem observe"  ← コメントアウト
  }
}
```

改善した場合は、claude-memの最新版にアップデートするか、代替手段としてClaude Code組み込みの `/memory` コマンドやプロジェクトメモリ（`CLAUDE.md` の `memory/` セクション）を活用しましょう。

## 即効性のある追加チューニング

### .claudeignoreで不要ファイルを除外

```bash
# プロジェクトルートに .claudeignore を作成
node_modules/
.next/
dist/
build/
*.lock
package-lock.json
*.min.js
*.map
.git/
coverage/
```

Next.jsプロジェクトの場合、`.next/` を除外するだけでコンテキスト消費が30〜40%削減されます。

### セッション分割戦略

1セッション = 1タスクを徹底します。40,000トークン × 3セッションは、180,000トークン × 1セッションより高速かつ高精度です。

```bash
# ❌ 1セッションで全部やろうとしない
"バグ修正して、テスト書いて、リファクタリングもして"

# ✅ タスクごとにセッションを分ける
# セッション1: "この関数のバグを修正して"
# /clear
# セッション2: "修正した関数のテストを書いて"
# /clear
# セッション3: "このモジュールをリファクタリングして"
```

### プランモードの活用

`Shift+Tab` でプランモードに切り替えると、Claudeはコード変更を実行せずに計画だけを立てます。試行錯誤による無駄なトークン消費を大幅に削減できます。

## 改善効果のまとめ

| 対策 | トークン削減効果 | 実施難易度 |
|------|:---:|:---:|
| CLAUDE.mdを50行以内に | 30〜40% | 低 |
| .claudeignore設定 | 25〜40% | 低 |
| /compact定期実行 | 20〜30% | 低 |
| セッション分割 | 40%+ | 中 |
| claude-mem競合の解消 | ケースによる | 中 |

これらを組み合わせることで、トータルで50%以上のトークン削減と、体感で60%程度の速度改善が見込めます。

## まとめ

Claude Codeが遅い・重いと感じたら、まず以下を確認してください：

1. **CLAUDE.mdは50行以内か？** → 超えていたらSkillファイルに分離
2. **コンテキスト使用量は70%未満か？** → `/context` で確認、`/compact` で圧縮
3. **claude-memなどのプラグインが競合していないか？** → 無効化して切り分け
4. **.claudeignoreは設定済みか？** → `node_modules/`、`.next/`、`dist/` を除外

再現性が低い問題だからこそ、原因の切り分け手順を知っておくことが重要です。この記事が、同じ問題で悩む開発者の時間を節約する一助になれば幸いです。
